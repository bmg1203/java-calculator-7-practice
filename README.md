## 개발 프로세스 및 규칙

### 1. 개발 사이클
**계획된 기능을 중심으로 Epic 생성**

**Epic에 해당하는 Story 생성**
- 이슈 생성 (at Jira)

**Story를 기준으로 기능 개발**
- 기능 구현 (code)

---

### 2. 이슈, 브랜치, 커밋 간의 관계

- **Epic ↔ 브랜치**
- **Story (이슈) ↔ 커밋, 기능 구현**

---

#### 이런 방법을 사용하는 이유

- 모든 작업이 **문서화되고 추적 가능**하며,
- **PR 생성 시 자동 연동**할 수 있음.
- 프로젝트를 계획하고 설계한 이슈 기준으로 **개발하고 관리**하는 것을 목표로 함.


> 현재 단계에서는 자동 연동은 구현되지 않음.

---

## 브랜치 네이밍 규칙

> <브랜치 종류> / <지라 이슈 키> + <설명>

### 브랜치 종류
- feature: 새로운 기능 개발 브랜치   
- fix: 버그 수정 브랜치
- hotfix: 긴급한 버그 수정
- refactor: 리팩토링 작업 브랜치
- chore: 설정 파일 수정, 문서 작업 등 비기능적 작업

### 예시
> feature/USER-12-login  
- 지라 이슈번호 12번의 유저 로그인 기능 개발 브랜치
> chore/gredle
- 그래들 설정 추가
---

## 커밋 네이밍 규칙

> <커밋 종류> : <기능 개발 내용>

### 커밋 종류
- feat: 새로운 기능 추가
- fix: 버그 수정
- hotfix: 긴급 버그 수정
- refact: 코드 리팩토링 (기능 변경 없이 코드 구조 개선)
- chore: 설정 파일 수정, 문서 작업 등 비기능적 작업
- test: 테스트 코드 추가/수정

### 예시
> feat: 로그인 기능 추가  
> fix: cors error 관련 문제 해결  
> refect: user 패키키지 구조

# 문자열 계산기 기능 명세

> **입력한 문자열에서 숫자 추출해 더하는 계산기**

## 예시
- **입력**:  
  `1+2:3`
- **결과값**:  
  `6`

---

## 기능 요구 사항 정리

1. 입력을 받아야 한다.
2. 입력 값에서 **숫자만 추출**해야 한다.
3. 추출한 숫자들을 **모두 더한다**.
4. 더한 결과를 **출력한다**.

---

## 기능 목록

| **기능 목록**                   | **구현**                                           |
|---------------------------------|----------------------------------------------------|
| 테스트 코드를 작성한다.         | 테스트 코드 작성 (초기엔 모두 실패해야 함)         |
| 입력을 받는다.                  | `Calculation` 클래스 정의, `Service` 클래스 정의, `getInputValues()` |
| 입력값이 올바른지 확인한다.      | `validateInput()`                                  |
| 에러를 구분하여 출력한다.        | `ExceptionHandler()`                               |
| 입력 값에서 숫자만 추출한다.     | `extractNumbersToList()`                           |
| 추출한 값을 더한다.              | `sumOfList()`                                      |
| 더한 결과값을 출력한다.          | `printResult()`                                    |

## 기능 구현 계획

--- 

### TDD (Test-Driven Development) 적용

#### TDD란?
- **테스트 주도 개발 방식**
- 테스트 코드를 먼저 작성 → 이후 개발하는 방식
- 초기 테스트 코드를 통과할 **최소한의 기능**을 개발

#### TDD를 사용하는 이유
- **테스트 코드를 먼저 작성**하게 될 경우, 여러 예외 사항을 미리 고려하기 때문에 **예측 범위 안에서 개발**이 가능하게 됨
- **모듈화**가 되기 때문에 → **유지 보수가 쉬운 코드**를 개발하게 됨
- **의도**: 테스트를 먼저 작성하기 때문에 원하는 기능을 **정확하게** 구현하게 됨
- 테스트를 진행하며 개발을 하기 때문에 → **어떤 부분에서 문제가 있는지 트래킹**하기 수월해짐

---

### 1. **Calculation 객체 설계**
- **Calculation 객체**는 현재 값을 저장 및 관리
- **필드 정의**:
   - `rawValue` : 입력받은 원본 문자열
   - `processedValue` : 숫자만 추출된 리스트

### 2. **Service 클래스 활용**
- `Calculation` 객체는 여러 **Service**에 정의된 함수를 통해 로직 처리
- 컨트롤 함수에서 이를 실행

### 3. **Controller 클래스 활용**
- `Service` 객체에 정의된 로직을 `Calculation` 에 적용

## 테스트 코드 작성

### **테스트 토픽**
- 입력 받기
- 입력값 유효성 확인
- 에러 구분 및 출력
- 입력값에서 숫자 추출
- 추출한 값 더하기
- 최종 결과 출력

---

## 기능별 상세 설계

### 1. **입력 기능**
- **토픽**: 입력 기능 개발
- **스토리**:
   - 문자열을 입력받는다.
   - 문자열의 유효성을 검증한다.

#### **유효성 검증 기준**
1. **잘못된 입력 처리**:
   - 숫자 + 구분자 + 숫자 형태로 입력이 주어져야 한다.
   - 숫자는 **양수**여야 한다.
   - **공백**: 현재는 공백을 무시함.

2. **커스텀 구분자 처리**:
   - `//` + … + `\n` 형태로 입력을 받음.
   - **커스텀 구분자 개수**: 현재는 한 개로 가정하고 개발.

---

### 2. **에러 핸들링 기능**
- **토픽**
  - 에러 핸들링 기능 개발
- **스토리**
   - 에러의 종류를 구분해 적절한 메시지를 출력한다.
   - V1에서는 **한 가지 에러**만 처리하지만, **확장성**을 고려해 여러 에러 핸들링 함수를 준비한다.

---

### 3. **숫자 추출 기능**
- **토픽**
  - 숫자 추출 기능 개발
- **스토리**
   - 입력받은 문자열에서 **숫자만 추출**해 리스트로 반환한다.

---

### 4. **값 더하기 기능**
- **토픽**
  - 값 더하기 기능 개발
- **스토리**
   - 추출된 숫자 리스트의 모든 요소를 더해 결과를 반환한다.

---

### 5. **출력 기능**
- **토픽**
  - 출력 기능 개발
- **스토리**
   - 최종 결과값을 **출력**한다.
